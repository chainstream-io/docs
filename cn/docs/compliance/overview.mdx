description: "与数据接口完全同源的 Know-Your-Transaction 与 Know-Your-Address 能力。"
icon: "shield-halved"
---

合规检查和数据本身在同一个 ChainStream API 里——不需要第二家供应商、第二张发票、第二份数据出域。给某一笔转账打风险分、给某个地址做画像、给某笔提现做放款前复核、导出 Travel Rule 所需的摘要——都在你现有的那套集成里完成。

## 什么时候需要它

<AccordionGroup>
  <Accordion title="交易所入金筛查">
    所有入金在入账前跑一遍 KYT，命中制裁、Mixer、暗网市场则拦截或冻结。
  </Accordion>
  <Accordion title="交易所出金筛查">
    提现场景用专用接口拿到欺诈评估分数与对手方地址的网络归属分解，放款前看一眼。
  </Accordion>
  <Accordion title="钱包入口风险">
    钱包入口在允许 swap、质押、存入之前，先对对手方地址做画像。
  </Accordion>
  <Accordion title="Travel Rule 报告">
    按监管要求导出每笔转账的摘要，字段与 FATF Travel Rule 报文对齐。
  </Accordion>
  <Accordion title="DeFi 前端尽责">
    签名前提示用户：对手方池子、代币创建者、收款地址的风险等级是多少。
  </Accordion>
</AccordionGroup>

## 合规怎么和数据拼起来

KYT 与 KYA 本质上就是另一类数据产品——查询方式和 tokens、trades、wallets 完全一致。这带来几个实用效果：

- 可以把 KYT 结果挂到你已有 **Trades / Transfers** 流水的每一行上
- **Wallets / Smart Money** 结果可以直接 join 上 KYA 实体标签
- 同一个 **GraphQL** 查询里就能把业务数据和合规过滤写在一起
- **Kafka** 转账主题可以和 REST 风险评分并行消费
