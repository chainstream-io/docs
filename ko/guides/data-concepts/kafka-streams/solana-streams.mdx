Solana는 400ms 블록 타임을 목표로 하며, 실제 처리량은 ~4,000 TPS이고 이론적 피크는 65,000 TPS입니다. 매우 높은 메시지 볼륨은 컨슈머 처리 용량에 높은 요구사항을 부과합니다.

**스키마 리포지토리**: [github.com/chainstream-io/streaming_protobuf/solana](https://github.com/chainstream-io/streaming_protobuf/tree/main/solana)

---

## 메시지 유형 개요

Solana Streams는 다음 메시지 유형을 제공합니다:

| 메시지 유형 | 설명 | Topic |
|:---|:---|:---|
| TradeEvents | DEX 거래 이벤트 | `sol.dex.trades` |
| TokenEvents | 토큰 이벤트 | `sol.tokens` |
| BalanceEvents | 잔액 변경 이벤트 | `sol.balances` |
| DexPoolEvents | 유동성 풀 이벤트 | `sol.dex.pools` |
| TransferEvents | 전송 이벤트 | `sol.transfers` |
| CandlestickEvents | 캔들스틱 데이터 | `sol.candlesticks` |

---

## 블록 수준 데이터

Solana는 기존의 블록 번호 대신 Slot을 타임라인 식별자로 사용합니다.

| 필드 | 타입 | 설명 |
|:---|:---|:---|
| `Slot` | uint64 | 슬롯 번호 (주요 식별자) |
| `BlockHeight` | uint64 | 블록 높이 |
| `BlockHash` | bytes | 블록 해시 |
| `ParentSlot` | uint64 | 부모 슬롯 |
| `Timestamp` | Timestamp | 블록 시간 |

<Warning>
Solana의 shred 스트림에서 Block Header가 불완전할 수 있으며, **Slot** 필드만 정확성이 보장됩니다.
</Warning>

## 트랜잭션 수준 데이터

### 핵심 트랜잭션 필드

| 필드 | 타입 | 설명 |
|:---|:---|:---|
| `Signature` | bytes | 트랜잭션 서명 (고유 식별자) |
| `Status` | Status | 실행 상태 |
| `Header` | TransactionHeader | 메타데이터 (수수료 및 서명자 포함) |
| `Index` | uint32 | 블록 내 위치 |

### Instructions

트랜잭션은 여러 Instructions를 포함하며, 이는 Solana 실행 모델의 핵심입니다:

| 필드 | 설명 |
|:---|:---|
| `ProgramAccountIndex` | 호출된 프로그램 |
| `Data` | 인코딩된 인스트럭션 데이터 |
| `AccountIndexes` | 참조된 계정 |
| `BalanceUpdates` | 이 인스트럭션으로 인한 SOL 잔액 변경 |

<Note>
하나의 Solana 트랜잭션에 여러 Instructions가 포함될 수 있으며, 각각 다른 프로그램을 호출합니다.
</Note>

## 전송 데이터

`TransferEvents`는 Solana 전송 정보를 제공합니다 (Topic: `sol.transfers`).

### TransferEvent 구조

```protobuf
message TransferEvent {
  Instruction instruction = 1;
  Block block = 2;
  Transaction transaction = 3;
  DApp d_app = 4;
  Transfer transfer = 100;
  TransferProcessed transfer_processed = 200;  // processed topic에 포함
}
```

### Transfer 핵심 필드

| 필드 | 타입 | 설명 |
|:---|:---|:---|
| `sender_token_account_address` | string | 발신자 토큰 계정 주소 |
| `sender_account_owner_address` | string | 발신자 계정 소유자 주소 |
| `receiver_token_account_address` | string | 수신자 토큰 계정 주소 |
| `receiver_account_owner_address` | string | 수신자 계정 소유자 주소 |
| `token_address` | string | 토큰 주소 (Mint) |
| `token_amount` | string | 전송 금액 |

### 토큰 메타데이터 (TokenEvent)

토큰 이벤트는 공통 `TokenEvents` (Topic: `sol.tokens`)를 사용하며:

| 필드 | 설명 |
|:---|:---|
| `address` | 토큰 주소 |
| `name` / `symbol` | 이름 및 심볼 |
| `decimals` | 소수점 자릿수 |
| `uri` | 메타데이터 URI |
| `metadata_address` | 메타데이터 주소 |
| `creators` | 생성자 목록 (주소, 검증 상태, 지분) |

### SolanaExtra 전용 필드

Solana 토큰은 추가 메타데이터 필드를 포함합니다:

| 필드 | 설명 |
|:---|:---|
| `collection_address` | 컬렉션 주소 (NFT) |
| `fungible` | 대체 가능 여부 |
| `is_mutable` | 변경 가능 여부 |
| `is_native` | 네이티브 SOL 여부 |
| `program_address` | 프로그램 주소 |
| `seller_fee_basis_points` | 생성자 로열티 기본 포인트 (NFT) |
| `token_standard` | 토큰 표준 |
| `mint_authority` / `freeze_authority` / `update_authority` | 권한 계정 |
| `is_verified_collection` | 검증된 컬렉션 여부 |

### 잔액 업데이트 계층 메커니즘 (Solana 전용)

이것은 Solana 스트림 데이터의 중요한 특성으로, 잔액 업데이트가 두 수준에서 제공됩니다:

<Tabs>
  <Tab title="인스트럭션 수준 잔액 업데이트">
    각 인스트럭션 실행 직후의 잔액 변경으로, 단일 단계 작업의 직접적인 영향을 반영합니다.

    ```
    Transaction
    └── Instruction 1
        └── BalanceUpdate: +100 SOL
    └── Instruction 2
        └── BalanceUpdate: -50 SOL
    └── Instruction 3
        └── BalanceUpdate: +25 SOL
    ```
  </Tab>
  <Tab title="트랜잭션 수준 잔액 업데이트">
    트랜잭션의 모든 인스트럭션 완료 후 최종 잔액 상태.

    ```
    Transaction
    └── FinalBalanceUpdate: +75 SOL (순 변경)
    ```
  </Tab>
</Tabs>

이를 통해 개발자가 세밀한 자금 흐름과 최종 상태를 모두 추적할 수 있습니다.

## DEX 데이터

`TradeEvents`는 DEX 거래 데이터를 제공하며 (Topic: `sol.dex.trades`), 공통 TradeEvent 구조를 사용합니다.

### Trade 핵심 필드

| 필드 | 설명 |
|:---|:---|
| `token_a_address` / `token_b_address` | 거래쌍 토큰 주소 |
| `user_a_amount` / `user_b_amount` | 사용자 거래 금액 |
| `pool_address` | 풀 주소 |
| `vault_a` / `vault_b` | 풀 볼트 주소 |
| `vault_a_amount` / `vault_b_amount` | 볼트 금액 |
| `bonding_curve` | 본딩 커브 정보 (해당되는 경우) |

### DApp 정보

| 필드 | 설명 |
|:---|:---|
| `program_address` | DEX 프로그램 주소 (예: Raydium, Orca) |
| `inner_program_address` | 내부 프로그램 주소 |
| `chain` | 체인 식별자 (CHAIN_SOLANA) |

### DexPoolEvent - 유동성 풀

| 필드 | 설명 |
|:---|:---|
| `type` | 이벤트 유형 (INITIALIZE/INCREASE_LIQUIDITY/DECREASE_LIQUIDITY/SWAP) |
| `address` | 풀 주소 |
| `token_a_address` / `token_b_address` | 토큰 주소 |
| `token_a_amount` / `token_b_amount` | 토큰 금액 |
| `lp_wallet` | LP 지갑 주소 |

## Solana 체인 특성

### Slot 메커니즘

Solana는 기존의 블록 번호 대신 Slot을 타임라인 식별자로 사용합니다:

| 특성 | 설명 |
|:---|:---|
| 블록 간격 | ~400ms |
| 메시지 볼륨 | EVM 체인보다 훨씬 많음 |
| 주요 식별자 | 슬롯 번호 |

### 메시지 패키징

트랜잭션은 소규모 배치로 패키징되며, 각 Kafka 메시지에는 최대 250개 트랜잭션이 포함됩니다.

### Block Header 완전성

<Warning>
Solana의 shred 스트림에서 Block Header가 불완전할 수 있으며, **Slot** 필드만 정확성이 보장됩니다. 다른 필드는 비어 있거나 부정확할 수 있습니다.
</Warning>

### 고처리량 처리 권장사항

Solana의 매우 높은 처리량 때문에 다음을 추천합니다:

1. **충분한 처리 용량**: 컨슈머에 충분한 CPU와 메모리 확보
2. **병렬 처리**: 멀티스레딩/코루틴으로 메시지 병렬 처리
3. **효율적인 파싱**: Protobuf 파싱 코드 최적화
4. **배치 쓰기**: 영속성이 필요한 경우 배치 데이터베이스 쓰기 사용

## Topic → 메시지 유형 매핑

| Topic | Proto 파일 | 메시지 유형 | 설명 |
|:---|:---|:---|:---|
| `sol.dex.trades` | trade_event.proto | TradeEvents | DEX 거래 이벤트 |
| `sol.dex.trades.processed` | trade_event.proto | TradeEvents | USD 가격, 의심 플래그 포함 |
| `sol.tokens` | token_event.proto | TokenEvents | 토큰 이벤트 |
| `sol.tokens.processed` | token_event.proto | TokenEvents | 설명, 이미지, 소셜 링크 포함 |
| `sol.balances` | balance_event.proto | BalanceEvents | 잔액 변경 이벤트 |
| `sol.transfers` | solana/transfer_event.proto | TransferEvents | Solana 전송 이벤트 |
| `sol.transfers.processed` | solana/transfer_processed_event.proto | TransferProcessedEvents | USD 가치 포함 |
| `sol.dex.pools` | dex_pool_event.proto | DexPoolEvents | 유동성 풀 이벤트 |
| `sol.candlesticks` | candlestick.proto | CandlestickEvents | 캔들스틱 데이터 |

## 코드 예시

### Python 예시: Solana DEX 거래 소비

```python
from kafka import KafkaConsumer
from common import trade_event_pb2  # streaming_protobuf 리포지토리에서 가져오기

# 컨슈머 생성
consumer = KafkaConsumer(
    'sol.dex.trades',
    bootstrap_servers=['<your_broker_address>'],
    security_protocol='SASL_SSL',
    sasl_mechanism='SCRAM-SHA-512',
    sasl_plain_username='your_username',
    sasl_plain_password='your_password',
    auto_offset_reset='latest',
    enable_auto_commit=False,
    group_id='my-solana-consumer'
)

# 메시지 소비 및 파싱
for message in consumer:
    trade_events = trade_event_pb2.TradeEvents()
    trade_events.ParseFromString(message.value)
    
    for event in trade_events.events:
        print(f"Pool: {event.trade.pool_address}")
        print(f"Token A: {event.trade.token_a_address}")
        print(f"Token B: {event.trade.token_b_address}")
        print(f"Amount A: {event.trade.user_a_amount}")
        print(f"Amount B: {event.trade.user_b_amount}")
        print(f"Slot: {event.block.slot}")
        print(f"DEX Program: {event.d_app.program_address}")
        print("---")
```

### Go 예시: 고성능 소비

```go
package main

import (
    "context"
    "log"

    "github.com/segmentio/kafka-go"
    "github.com/segmentio/kafka-go/sasl/scram"
    "google.golang.org/protobuf/proto"
    
    common "github.com/chainstream-io/streaming_protobuf/common/messages"
)

func main() {
    mechanism, _ := scram.Mechanism(scram.SHA512, "your_username", "your_password")
    
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers: []string{"<your_broker_address>"},
        Topic:   "sol.dex.trades",
        GroupID: "my-solana-consumer",
        Dialer: &kafka.Dialer{
            SASLMechanism: mechanism,
            TLS:           &tls.Config{},
        },
    })
    defer reader.Close()

    // 워커 풀로 병렬 처리
    messages := make(chan kafka.Message, 100)
    
    // 워커 시작
    for i := 0; i < 10; i++ {
        go func() {
            for msg := range messages {
                var tradeEvents common.TradeEvents
                if err := proto.Unmarshal(msg.Value, &tradeEvents); err != nil {
                    log.Printf("Failed to parse: %v", err)
                    continue
                }
                
                // 거래 처리
                for _, event := range tradeEvents.Events {
                    log.Printf("Pool: %s, Token A: %s, Token B: %s",
                        event.Trade.PoolAddress,
                        event.Trade.TokenAAddress,
                        event.Trade.TokenBAddress)
                    log.Printf("Amount A: %s, Amount B: %s, Slot: %d",
                        event.Trade.UserAAmount,
                        event.Trade.UserBAmount,
                        event.Block.Slot)
                }
            }
        }()
    }

    // 메시지 읽기
    for {
        msg, err := reader.ReadMessage(context.Background())
        if err != nil {
            log.Printf("Error reading: %v", err)
            continue
        }
        messages <- msg
    }
}
```
