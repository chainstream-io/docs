合規檢查和資料本身在同一個 ChainStream API 裡——不需要第二家供應商、第二張發票、第二份資料出域。給某一筆轉賬打風險分、給某個地址做畫像、給某筆提現做放款前複核、匯出 Travel Rule 所需的摘要——都在你現有的那套整合裡完成。

## 什麼時候需要它

<AccordionGroup>
  <Accordion title="交易所入金篩查">
    所有入金在入賬前跑一遍 KYT，命中制裁、Mixer、暗網市場則攔截或凍結。
  </Accordion>
  <Accordion title="交易所出金篩查">
    提現場景用專用介面拿到欺詐評估分數與對手方地址的網路歸屬分解，放款前看一眼。
  </Accordion>
  <Accordion title="錢包入口風險">
    錢包入口在允許 swap、質押、存入之前，先對對手方地址做畫像。
  </Accordion>
  <Accordion title="Travel Rule 報告">
    按監管要求匯出每筆轉賬的摘要，欄位與 FATF Travel Rule 報文對齊。
  </Accordion>
  <Accordion title="DeFi 前端盡責">
    簽名前提示使用者：對手方池子、代幣建立者、收款地址的風險等級是多少。
  </Accordion>
</AccordionGroup>

## 合規怎麼和資料拼起來

KYT 與 KYA 本質上就是另一類資料產品——查詢方式和 tokens、trades、wallets 完全一致。這帶來幾個實用效果：

- 可以把 KYT 結果掛到你已有 **Trades / Transfers** 流水的每一行上
- **Wallets / Smart Money** 結果可以直接 join 上 KYA 實體標籤
- 同一個 **GraphQL** 查詢裡就能把業務資料和合規過濾寫在一起
- **Kafka** 轉賬主題可以和 REST 風險評分並行消費
