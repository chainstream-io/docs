## 概覽

每筆 GraphQL 查詢都會依查詢複雜度動態計算並消耗 **Credit Unit（CU，點數單位）**。點數從與 REST API **相同**的計費方案扣除——同一把 API Key 兩邊通用。

<Info>
GraphQL API 與 REST Data API 共用同一組 API Key 與計費方案。GraphQL 查詢消耗的點數會計入整體用量。
</Info>

---

## 點數計算公式

點數依 Cube 以 **五因子公式**計算。最終 CU 取決於你查哪些 Cube、要幾列、是否彙總、納入幾個 metric，以及選了幾個欄位。

```
CU = ceil(BaseCost × LimitFactor × AggregationFactor × MetricFactor × ComplexityFactor) / 100
```

| 因子 | 計算方式 | 說明 |
|:-------|:-----------|:------------|
| **BaseCost** | 各 Cube 內部值 | 反映底層表大小與掃描成本。每個 Cube 的 base cost 不同。 |
| **LimitFactor** | `ceil(limit / 100)`，至少為 1 | 與請求的列數線性成長。1–100 列成本相同；101–200 為 2 倍，以此類推。 |
| **AggregationFactor** | 1.0（無）、1.5（GROUP BY）、2.0（HAVING） | 使用彙總或彙總後篩選（`selectWhere`）時較高。 |
| **MetricFactor** | `1.0 + (aggregate_count × 0.1)` | 每多一個彙總函式（`count`、`sum`、`avg` 等）成本約增加 10%。 |
| **ComplexityFactor** | `min(1.0 + min(select_count, 250) / 50 × 0.2, 1.5)` | 選越多欄位成本越高，上限 1.5×。約 5 個欄位的簡單查詢因子約 1.02。 |

<Note>
**零列查詢免費。** 若查詢未回傳任何列，不論其他因子為何都不扣點。
</Note>

### 計算範例

<AccordionGroup>
  <Accordion title="簡單查詢：10 列、5 個欄位">
    ```
    BaseCost = 2000 (internal), LimitFactor = ceil(10/100) = 1
    AggregationFactor = 1.0, MetricFactor = 1.0
    ComplexityFactor = 1.0 + (5/50) × 0.2 = 1.02
    Internal = ceil(2000 × 1 × 1.0 × 1.0 × 1.02) = 2040
    CU = 2040 / 100 = 20.40 CU
    ```
  </Accordion>
  <Accordion title="大型查詢：500 列、5 個欄位">
    ```
    BaseCost = 2000, LimitFactor = ceil(500/100) = 5
    AggregationFactor = 1.0, MetricFactor = 1.0
    ComplexityFactor = 1.02
    Internal = ceil(2000 × 5 × 1.0 × 1.0 × 1.02) = 10200
    CU = 10200 / 100 = 102.00 CU
    ```
  </Accordion>
  <Accordion title="彙總查詢：GROUP BY + 2 個 metric、500 列">
    ```
    BaseCost = 2000, LimitFactor = ceil(500/100) = 5
    AggregationFactor = 1.5 (GROUP BY)
    MetricFactor = 1.0 + 2 × 0.1 = 1.2
    ComplexityFactor = 1.02
    Internal = ceil(2000 × 5 × 1.5 × 1.2 × 1.02) = 18360
    CU = 18360 / 100 = 183.60 CU
    ```
  </Accordion>
  <Accordion title="複雜查詢：選取大量欄位">
    ```
    BaseCost = 2000, LimitFactor = 1
    AggregationFactor = 1.0, MetricFactor = 1.0
    ComplexityFactor = min(1.0 + (250/50) × 0.2, 1.5) = 1.5 (capped)
    Internal = ceil(2000 × 1 × 1.0 × 1.0 × 1.5) = 3000
    CU = 3000 / 100 = 30.00 CU
    ```
  </Accordion>
</AccordionGroup>

<Tip>
由於 CU 為動態計算，要知道確切成本，最可靠的方式是查看回應中的 `extensions.credits` 欄位，或監看 IDE 狀態列上的 CU 指示。
</Tip>

## 回應：`extensions.credits`

每筆 GraphQL 回應都會在 `extensions` 中附上點數消耗資訊：

```json
{
  "data": {
    "Solana": {
      "DEXTrades": [ ... ]
    }
  },
  "extensions": {
    "credits": {
      "total": 20.4,
      "unit": "CU",
      "cubes": [
        {
          "cube": "DEXTrades",
          "credits": 20.4,
          "row_count": 10
        }
      ]
    }
  }
}
```

| 欄位 | 型別 | 說明 |
|:------|:-----|:------------|
| `total` | `Float` | 整筆查詢消耗的 CU 總計 |
| `unit` | `String` | 固定為 `"CU"` |
| `cubes` | `Array` | 依 Cube 拆分 |
| `cubes[].cube` | `String` | 計費引擎使用的 Cube 名稱 |
| `cubes[].credits` | `Float` | 該 Cube 計收的 CU |
| `cubes[].row_count` | `Int` | 回傳列數 |

<Note>
有扣點時（即 `total > 0`）才會出現 `extensions.credits`。零列查詢不計費。
</Note>

## 在 IDE 中監控用量

[GraphQL IDE](https://ide.chainstream.io) 狀態列會在每筆查詢後顯示點數消耗：

- **CU 指示**：顯示該次查詢消耗的 CU 總量
- **Latency**：請求耗時（毫秒）
- **Response size**：回應本文大小

## 優化點數使用的建議

<CardGroup cols={3}>
  <Card title="少選欄位" icon="minimize">
    只請求需要的維度。ComplexityFactor 會隨選取欄位數增加。
  </Card>
  <Card title="適當的 limit" icon="gauge">
    在可接受範圍內盡量降低 `limit.count`。LimitFactor 每多 100 列會加倍。
  </Card>
  <Card title="善用預先彙總 Cube" icon="layer-group">
    需要彙總資料時，優先使用 DWM／DWS Cube（Pairs、Tokens、TokenHolders），而非在 DWD Cube（DEXTrades）上跑 metric。
  </Card>
</CardGroup>
