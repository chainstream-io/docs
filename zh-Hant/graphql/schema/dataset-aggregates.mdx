## 概覽

ChainStream GraphQL 中，每個 Chain Group 都可接受兩個選填參數，用來控制**實際查詢的底層表**。可依情境在資料新鮮度、查詢速度與資料完整度之間取捨。

---

## `dataset` 參數

`dataset` 參數控制所查資料的**時間範圍**。它決定查詢會打到即時表、歸檔表，或兩者皆查。

| 值 | 說明 | 典型用途 |
|:------|:------------|:-----------------|
| `combined` | 同時查即時與歸檔資料 **（預設）** — 通常涵蓋最近約 7–10 天 | 需要完整時間範圍的通用查詢 |
| `realtime` | 僅近期資料（約最近 24 小時內） | 監控儀表板、最新成交、即時告警 |
| `archive` | 僅保留區間內的歷史資料（約 7–10 天） | 歷史分析、回填、趨勢研究 |

### 用法

```graphql
query {
  Solana(dataset: realtime) {
    DEXTrades(limit: {count: 10}, orderBy: {descending: Block_Time}) {
      Block { Time }
      Trade { Buy { Currency { MintAddress } Amount PriceInUSD } }
    }
  }
}
```

```graphql
query {
  EVM(network: eth, dataset: archive) {
    Transfers(
      where: { Block: { Time: { after: "2026-01-01T00:00:00Z", before: "2026-02-01T00:00:00Z" } } }
      limit: {count: 100}
    ) {
      Block { Time }
      Transfer { Currency { MintAddress } Amount AmountInUSD }
    }
  }
}
```

### 歷史資料回填

建置資料管線或從中斷恢復時，可搭配時間範圍篩選使用 `dataset: archive` 回填歷史：

1. 記錄最後處理的時間戳或區塊高度
2. 以 `dataset: archive` 搭配 `where`，從上次檢查點查到目前時間
3. 處理回填資料
4. 改為 `dataset: realtime` 做持續監控

```graphql
query BackfillTrades {
  Solana(dataset: archive) {
    DEXTrades(
      where: {
        Block: {
          Time: {
            after: "2026-04-01T00:00:00Z"
            before: "2026-04-02T00:00:00Z"
          }
        }
      }
      limit: {count: 10000}
      orderBy: {ascending: Block_Time}
    ) {
      Block { Time Slot }
      Transaction { Hash }
      Trade {
        Buy { Currency { MintAddress } Amount PriceInUSD }
        Sell { Currency { MintAddress } Amount }
      }
    }
  }
}
```

### 不支援 `dataset` 切換的表

部分 Cube 不論 `dataset` 為何都查同一張表，包括：

- **DWS Cube**：`TokenHolders`、`WalletTokenPnL`、`DEXPools` — 代表目前狀態快照
- **特殊表**：`TransactionBalances`、`PredictionTrades`、`PredictionManagements`、`PredictionSettlements`

對這些 Cube，`dataset` 會被靜默忽略。

## `aggregates` 參數

`aggregates` 參數控制查詢是否使用**預先彙總的 materialized view**（DWM 層），而非原始明細表（DWD 層）。預先彙總表內含預先計算的 rollup（通常為每分鐘），查詢會快很多。

| 值 | 說明 | 典型用途 |
|:------|:------------|:-----------------|
| `yes` | 在可用時優先使用預先彙總表 **（預設行為）** | 多數分析查詢 |
| `no` | 僅使用原始明細表 | 需要單筆事件粒度時 |
| `only` | 僅使用預先彙總表 | 追求最快速度，可接受較少欄位 |

### 用法

```graphql
query {
  EVM(network: eth, aggregates: only) {
    Pairs(
      where: { Token: { Address: { is: "0xdac17f958d2ee523a2206206994597c13d831ec7" } } }
      limit: {count: 100}
      orderBy: {descending: Block_Time}
    ) {
      Interval { Time }
      Price { Ohlc { Open High Low Close } }
      Volume { Usd }
    }
  }
}
```

### 各模式何時使用

| 情境 | 建議 | 原因 |
|:---------|:------------|:----|
| 建立 OHLC 圖表 | `aggregates: only` | 預先計算的 K 線資料，回應最快 |
| 隨時間的成交量趨勢 | `aggregates: yes` | 善用預先彙總的成交量統計 |
| 單筆成交分析 | `aggregates: no` | 需要 rollup 無法提供的單筆明細 |
| 計算不重複交易者數 | `aggregates: yes` | 可使用預先計算的不重複計數 |

## 兩者並用

`dataset` 與 `aggregates` 可同時使用：

```graphql
query {
  Trading(dataset: realtime, aggregates: yes) {
    Tokens(
      where: { Token: { Address: { is: "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v" } } }
      limit: {count: 60}
      orderBy: {descending: Block_Time}
    ) {
      Interval { Time }
      Volume { Usd BuyVolumeUSD SellVolumeUSD }
      Stats { TradeCount UniqueBuyers UniqueSellers }
    }
  }
}
```

此查詢以即時資料搭配預先彙總表，取得最近約 60 分鐘的跨鏈代幣交易統計，以追求較快速度。

## 效能考量

<CardGroup cols={2}>
  <Card title="儀表板請用 realtime" icon="gauge-high">
    `dataset: realtime` 會打到較小的表分割區，監控情境下回應較快。
  </Card>
  <Card title="分析請用 aggregates" icon="chart-line">
    `aggregates: yes` 或 `only` 會使用預先計算的 rollup，比掃描原始事件表快上數個數量級。
  </Card>
</CardGroup>

<Tip>
若要 OHLC 或成交量查詢盡可能快，可同時使用 `dataset: realtime` 與 `aggregates: only`，鎖定最小、最優化的資料切片。
</Tip>
